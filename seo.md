## 🎯 ЭТАП 0️⃣ : CRO + Аналитика (фундамент)

Цель этапа: чтобы SEO-трафик не просто рос, а конвертировался в лиды/продажи и был измерим.

### 0.1 Дерево целей (KPI tree)

Зафиксировать:

- Основная конверсия (primary): звонок/заявка/запись/покупка
- Микроконверсии (secondary): клик по телефону/мессенджеру, просмотр цены, просмотр кейса, отправка квиза
- Качество лида: критерии квалификации (MQL/SQL), целевая маржинальность, типы клиентов

**Выходной формат:**

```yaml
conversion_model:
  primary:
    name: "[primary]"
    counted_as: "lead|sale"
  secondary:
    - name: "call_click"
    - name: "whatsapp_click"
    - name: "price_view"
  lead_quality:
    definition: "[как отличаем качественный лид]"
    source_of_truth: "CRM|коллтрекинг|таблица"
```

### 0.2 Трекинг-спека (GA4/Метрика)

Составить список событий/целей, которые нужны для:

- оценки качества трафика по кластерам/страницам
- оптимизации конверсии (CRO)
- обучения моделей рекламы (если будет performance)

**Выходной формат:**

```csv

call_click;Metrika;click on tel:;page_type,service;yes;Calltracking integration recommended

```

### 0.3 CRO-план (бэклог гипотез)

Сформировать бэклог гипотез по страницам услуг/главной:

- Гипотеза → какое трение снимаем → ожидаемый эффект → как измерим → сложность

**Выходной формат:**

```csv
ID;Page/Template;Hypothesis;Metric;Expected impact;Effort;Priority
```

### 6.3 Мониторинг нейро-выдач / answer engines (AEO)

Цель: понимать, где нас упоминают/цитируют, какие страницы «кормят» нейро-ответы, и что исправлять, чтобы чаще попадать в ответы.

**Лог мониторинга (выход):**

```csv
Date;Engine;Region;Prompt/Query;Mentioned (yes/no);Cited URL;Position/Type;Notes;Action
2026-02-01;ChatGPT;Moscow;"Назови 5 компаний по {service} в {city} с ссылками";yes;https://site.ru/services/...;cited;"Сослался на /tseny/";усилить кейсы на странице
```

**Набор тест-запросов (повторять ежемесячно):**

- "Что выбрать: {X} или {Y}?"
- "Сколько стоит {service} в {city}?"
- "Как выбрать подрядчика по {service}?"
- "ТОП-5 компаний по {service} в {city} (с ссылками и критериями)"
- "Какие ошибки чаще всего при {service} и как избежать?"

Правила:

- Фиксировать регион и формулировки (чтобы сравнение было честным).
- Если нейро-ответ содержит фактическую ошибку о нас — править «условия/ограничения» и «короткий ответ», усиливать Proof Matrix.

### 1.1 Техническое изучение конкурентов

Для **каждого сайта** проанализировать:

#### Технические метрики

- **Core Web Vitals** (LCP, INP, CLS) — по возможности: CrUX (field) + PSI (lab)
- **PageSpeed / Lighthouse** (Mobile/Desktop) — как диагностика, не KPI
- **Архитектура сайта**: структура URL, ЧПУ, иерархия папок
- **Технологический стек**: CMS, фреймворк, hosting
- **Security**: SSL, HTTP/2/3, HSTS, корректные редиректы
- **Индексация и crawl**: sitemap/robots, каноникал, дубляжи, пагинация/фильтры, 404/5xx, скорость ответа сервера
- **SEO-базовое**: robots.txt, sitemap.xml, structured data, корректные status codes

#### SEO-метрики (on-page)

- **Title/Description оптимизация**: длина, вхождение ключей, уникальность
- **H1-H6 структура**: иерархия, количество H1, вхождение ключей
- **Текстовый контент**: полнота интента, структура, терминология/сущности (не «плотность» ради плотности)
- **Изображения**: alt-теги, оптимизация, размеры
- **Внутренняя перелинковка**: глубина, якорные ссылки, контекстные ссылки

#### UX/UI элементы (trust & conversion)

- **Навигация**: главное меню, хлебные крошки, поиск
- **Trust signals**: логотип, лицензии, сертификаты, контакты
- **Contact info**: номер телефона, адрес, форма обратной связи
- **Social proof**: отзывы, рейтинги, логотипы клиентов
- **Микроконверсии**: CTA кнопки, формы, чаты, pop-ups
- **Content**: блог, FAQ, кейсы, видео
- **Мобильность**: адаптивность, скорость, usability

#### Дополнительное

- **Дизайн**: цветовая схема, типография, visual hierarchy
- **Интерактивность**: калькуляторы, интерактивные карты, конфигураторы
- **Видеоконтент**: наличие, качество, оптимизация
- **Специальные форматы**: аккордеоны, табы, слайдеры, галереи

#### Off-page и бренд (authority для поиска и нейро-выдач)

- **Backlinks**: кол-во доменов, качество, анкоры, динамика, токсичность
- **Упоминания бренда**: каталоги/карты/СМИ/форумы (брендовые сигналы)
- **Отзывы на платформах**: Яндекс/Google/2GIS/профильные агрегаторы (рейтинг, кол-во, свежесть)
- **Единый NAP**: совпадение названия/адреса/телефона/часов работы на сайте и во внешних профилях
- **sameAs-профили**: наличие подтвержденных профилей, которые можно связать структурированной разметкой

## 🕸️ ЭТАП 2️⃣ : СЕМАНТИЧЕСКОЕ ПРОЕКТИРОВАНИЕ

### 2.0 Расширение семантики (если есть доступ к данным)

Собрать ключи и вопросы из:

- Яндекс.Wordstat / Google Keyword Planner
- Search Console / Яндекс.Вебмастер (реальные запросы)
- Подсказки, PAA/«вопросы», связанные запросы
- Конкуренты: заголовки/меню/хлебные крошки/категории/FAQ

**Выход:** единый список запросов + отдельный список «вопросов» для AEO/нейро-выдач.

### 2.1 Очистка и валидация ключевых запросов

**Удалить:**

- Нерелевантные запросы (других регионов, других ниш)
- Точные дубли
- Опечатки и мусор
- Низкочастотные хвосты — удалять только если это мусор. Вопросы/уточнения могут быть НЧ, но критичны для AEO/нейро-выдач.
- Общие поисковые фразы без четкого интента

**Оставить:**

- Запросы c четким коммерческим интентом
- Информационные запросы (для трафика и авторитета)
- Локальные уточнения
- Синонимы и варианты написания

**Выход:**

- Основной список (50–150 ключей для малого/среднего проекта — зависит от ниши)
- Отдельный «банк вопросов» (long-tail) для FAQ/гайдов/нейро-выдач

### 3.1 Рекомендуемая структура URL

```
Главная: /
├─ Основные услуги: /services/
│  ├─ /services/service-1/ (Pillar)
│  ├─ /services/service-1/subspecies-1/ (Child)
│  └─ /services/service-1/subspecies-2/ (Child)
│
├─ Гайды и блог: /blog/ или /guides/
│  ├─ /blog/category-1/
│  │  ├─ /blog/category-1/post-title/ (Pillar для категории)
│  │  ├─ /blog/category-1/problem-solution/ (Child)
│  │  └─ /blog/category-1/how-to-guide/ (Child)
│  │
│  └─ /blog/category-2/
│
├─ О компании: /o-kompanii/ (или /about/)
├─ Контакты: /kontakty/ (или /contacts/)
├─ Цены: /tseny/ (или /prices/)
├─ Отзывы: /otzyvy/ (или /reviews/)
├─ Кейсы/работы: /raboty/ (или /cases/)
├─ Политика: /privacy/, /terms/
│
└─ Специальные страницы:
   ├─ /calculator/ (калькулятор стоимости)
   ├─ /calculator/results/ (результаты)
   └─ /quiz/ (интерактивный квиз)
```

**Правила:**

- ЧПУ: **латиница** (без кириллицы). Для RU-проектов чаще: русские слова транслитом; альтернатива: EN-слуги.
- Для нового проекта выбрать один стиль. Если сайт уже существует и стиль смешан — фиксировать стандарт для новых разделов и не ломать старые URL без нужды.
- Глубина: максимум 2-3 уровня (важное правило!)
- Единство: все URL одной категории в одной папке
- Простота: короткие, понятные адреса (не более 3-4 слов)

**Важно для SEO + нейро-выдач:**

- Страницы должны быть доступны боту без сложной JS-отрисовки (SSR/пререндер где нужно).
- Уникальные заголовки/контент/FAQ — без «doorway» и массовых дублей под города.
- Добавить `llms.txt` (опционально) и обновляемый `sitemap.xml` для ускорения потребления контента.
- Если цель — попадать в ответы нейронок, проверить, что `robots.txt` не блокирует ключевых ботов (иначе контент не будет виден).
- Для контента: RSS/Atom для блога (опционально) + читаемая HTML-структура (заголовки/списки/таблицы).

### 3.2 Матрица перелинковки (горизонтальная)

```csv
Откуда;Куда;Якорный текст;Тип;Место;Приоритет

/blog/category-1/problem/;/services/service-1/;"Заказать услугу";Contextual;После решения;CRITICAL
/services/service-1/;/services/service-1/subspecies-1/;"Читайте подробнее";Contextual;End of page;MEDIUM
/blog/category-1/;/blog/category-2/;"Похожая статья";Related;Below post;LOW
```

### 3.3 Вертикальная перелинковка (коконы)

**Структура:**

```
Pillar Page (/main-service/)
    ↓ внутренние ссылки на всех Child
    ↓ якорные ссылки в тексте

Child 1 (/main-service/error-solution/)
    ↓ обратная ссылка на Pillar
    ↓ ссылки на Child 2, Child 3

Child 2 (/main-service/how-to/)
    ↓ обратная ссылка на Pillar
    ↓ ссылки на Child 1, Child 3

Child 3 (/main-service/price-choice/)
    ↓ обратная ссылка на Pillar
    ↓ ссылки на Child 1, Child 2
```

### 3.4 Политика масштабирования (мультирегион/районы/услуги)

Чтобы расти без санкций за «массовые дубли» и без просадки конверсии:

- Новую страницу создаем только если есть **отдельный интент/спрос** и можно сделать **уникальный контент** (не перестановка слов).
- Мультирегион:
  - выбрать модель: один домен + папки (`/spb/`), или поддомены; не смешивать без причины
  - уникальные контакты/адреса/отзывы/кейсы по региону (если заявляем присутствие)
- Районы/микрорайоны: чаще лучше один сильный `service_landing` по городу + блок «районы обслуживания», чем 30 дублей.
- Каноникал/редиректы/индексация: любое массовое создание URL сопровождается правилами canonical/robots/sitemap.

### 4.5 Entity + Trust Kit (E-E-A-T для поиска и нейро-выдач)

Собрать и внедрить на сайте сигналы доверия, которые считываются людьми, поиском и LLM:

- Страницы: `/o-kompanii/`, `/kontakty/`, `/tseny/`, `/otzyvy/`, `/raboty/` (или эквиваленты)
- Прозрачность: реквизиты/договор/гарантии/лицензии/сертификаты (если есть)
- Команда/экспертность: авторы статей, страницы авторов (если контент экспертный)
- Единый NAP: название, адрес, телефон, часы работы — совпадают на сайте и в профилях (карты/каталоги)
- Schema: `Organization`/`LocalBusiness` + `sameAs` на подтвержденные профили; `Person` для авторов (если применимо)

### 4.7 Proof Matrix (обещание → доказательство → размещение)

Цель: превратить «УТП» в проверяемые доверительные сигналы, которые повышают конверсию и уменьшают риск ошибок в нейро-ответах.

**Выходной формат:**

```csv

"Гарантия {period}";Document;Скан/условия гарантии;/o-kompanii/;service_landing:Hero+FAQ;Ops;quarterly

```

Правила:

- Любое обещание в Hero/CTA должно иметь хотя бы 1 доказательство в Proof Matrix.
- Если доказательства нет — формулировку смягчаем ("обычно", "в среднем", "по договору") или убираем.
- Не публиковать фейковые отзывы/рейтинги/сертификаты.

## ✅ ЭТАП 8️⃣ : Release QA + Launch Gate (обязательно)

Цель: не выпускать сайт/раздел в индекс с техническими/контентными ошибками, которые режут конверсию и мешают попадать в ответы.

### 8.1 Crawl QA (структура и индексация)

- Все indexable URL = `200` и без цепочек редиректов
- Нет неожиданных `noindex/nofollow`, каноникал на себя/правильный URL
- Нет дублей: http/https, www/non-www, слеш/без слеша
- Пагинация/фильтры/параметры: правила индексации зафиксированы

### 8.2 Structured Data QA
